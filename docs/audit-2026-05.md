# Audit — May 2026

> A retrospective on the `extract_source` regression that motivated v0.1.1.
> Published so future contributors understand why the function is shaped the way it is.

## Summary

Twelve cached audio files from the development vault were audited against their source notes. Of those, **7 contained narration that bore little or no relation to the source**:

- 3 were *complete drift*: the spoken audio regurgitated VOICE-corpus sample passages with zero source content (book gem narrations).
- 2 *hallucinated stories* that the source did not contain (archive notes, setup guide).
- 2 *abstracted away the specifics* the listener actually needed (registry of automations, aphorism index).

2 were faithful condensations of the source. 3 had minor framing issues. The pipeline had been silently substituting the VOICE.md contract (loaded as the system context on every call) for the actual source content whenever the extracted text was thin.

## Root cause

`extract_source(text, whole=False)` collected:

1. The `tldr:` value from frontmatter — **only that exact key**, missing `gist:`, `summary:`, `description:`.
2. The **labels** of `## H2` sections — not the body underneath.
3. Fell back to the whole body **only if** both above yielded zero text.

For any note with structural H2 labels (book gems with "Gist / Key principles / Memorable quotes / How to apply / Adjacent", a SETUP guide with "1. Mac / 2. GitHub remote / 3. Mobile", a distilled livre note in FR), the labels existed → the fallback never triggered → the LLM tightening step received an outline with no substance → it confabulated content using VOICE.md's own sample passages as the seed (the system prompt was the most concrete text available).

The audio sounded right because it WAS in the right voice. It was just narrating the wrong content.

## Fix

The function now:

1. Looks for `tldr` / `gist` / `summary` / `description` in frontmatter (first match wins, case-insensitive).
2. For each `## H2` section, captures the **heading + the first ~80 words** of body content. Skips code blocks, tables, separators.
3. Strips markdown noise (wikilinks, inline code, bold, italics, H3 markers, list markers, blockquotes, HTML).
4. Still falls back to the whole body if nothing was collected (e.g., notes that are pure prose with no headings or frontmatter).

The function is now covered by `.github/workflows/extract.yml` with 8 fixture cases asserting:

- Each frontmatter key alternative is recognized.
- H2 *body* (not label) makes it into the output.
- Wikilinks and inline code render as plain text.
- H3 markers don't bleed through as `###`.
- Whole-body fallback still works when nothing structural is present.
- `whole=True` bypasses extraction.

## Operational follow-ups

- **All stale audio cache cleared** from the development vault. Re-narration on next read uses the fixed extraction.
- **No change to the cache invalidation logic** — the existing mtime check works correctly; the bug was upstream of caching.
- **No change to the LLM tightening prompt** — the issue was the input, not the prompt.

## What this means for users

If you read a note aloud in v0.1.0 (the buggy version), the audio might be faithful or might not be — there was no way to know without comparing. **In v0.1.1, regenerate any note you care about.**

A note whose audio you suspect drifted: open the source, edit anything (even a trailing space), save. The cache will see the new mtime and regenerate on next read.

The audit tooling (`voice/audit_audio.py`) is included in the repo so you can run it yourself against any cached audio:

```bash
cd <your-vault>/.obsidian/plugins/bedrock-voice/voice
source .venv/bin/activate
python audit_audio.py
```

Flags: orphan source, ambiguous source, stale cache, cross-language (by design vs. drift), uncovered H2 sections, novel-token ratio.
